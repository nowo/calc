# 性能

`@wzo/calc` **0 运行时依赖**，ESM 产物 gzip 后 **~12 KB**，所有运算内部走 BigInt。
虽然比原生浮点慢（原生快但**会算错**），精度运算依然达到**每秒数百万次**，日常场景绰绰有余。

## 横向对比

与主流精度库的实测吞吐对比（ops/s，越高越快；数值因机器而异，仅供量级参考）：

### 单步运算

| 操作 | `@wzo/calc` | decimal.js | mathjs | a-calc |
| :--- | ---: | ---: | ---: | ---: |
| 加法 `0.1 + 0.2` | **9,090,000** 🥇 | 2,920,000 | 2,040,000 | 431,000 |
| 乘法 `1.1 * 2.2` | **8,160,000** 🥇 | 2,280,000 | 1,590,000 | 139,000 |
| 除法 `1 / 3` | 2,740,000 | **3,240,000** 🥇 | 2,040,000 | 450,000 |

- **加法 / 乘法**：`@wzo/calc` 最快，领先 decimal.js 约 **3～3.6 倍**。这两项走 **number 整数缩放快路径**——操作数都在 `2^53` 安全整数内、小数位 ≤ 15 时用 `number` 整数运算（接近原生速度），可能溢出才回退 BigInt；缩放只做整数 × 10ᵏ，绝不用浮点乘小数，所以无精度损失
- **除法**：decimal.js 仍最快（它的 base-1e7 长除法做了重度优化；且我们算 50 位**小数**，它算 50 位**有效数字**，工作量略不同）。借鉴它「预存常量、余数为 0 即提前终止」的思路——**预存 10 的幂**避免重复构造大整数、**整除走快路径**——我们已从约 170 万/s 提到 **~274 万/s**，差距从 1.8× 收窄到 **1.18×**

### 复杂表达式求值

真实业务里很少是单步运算，更多是含括号、折扣满减、连乘复利、多次除法的复合表达式。下面几条均已与 mathjs（BigNumber 80 位）交叉验证、结果**逐一吻合**，仅比吞吐（decimal.js 无表达式解析，不参与）：

| 表达式 | `@wzo/calc` | mathjs | a-calc | decimal.js |
| :--- | ---: | ---: | ---: | ---: |
| `1 + 2 * 2`（简单） | **2,310,000** 🥇 | 590,000 | 419,000 | —（无表达式解析）|
| 满减折扣 `(999.99*3 + 499.5*2 + 199.9*5)*0.85 - 200` | **463,000** 🥇 | 98,000 | 207,000 | — |
| 复利连乘 `100000 * 1.004⁶ - 100000` | **299,000** 🥇 | 148,000 | 198,000 | — |
| 多次除法 `1000/3 + 2000/7 + 3000/11 + 4000/13 + 5000/17`（循环小数）| **132,000** | 85,000 | 143,000 🥇 | — |

- **加减乘为主的表达式 `@wzo/calc` 最快**：简单表达式领先 a-calc 约 **5.5 倍**、mathjs 约 **3.9 倍**；满减折扣领先 a-calc 约 **2.2 倍**、mathjs 约 **4.7 倍**
- 除法占主导时（多次除法），优势收窄到与 a-calc **基本持平**（互有胜负），仍领先 mathjs 约 1.6 倍——高精度除法只能走 BigInt，吃不到 number 快路径
- 单步加减乘上 `@wzo/calc` 比同为字符串引擎的 **a-calc** 快 **19～20 倍**

> 对比基准在 `packages/main/tests/compare.bench.ts`，跑 `pnpm --filter @wzo/calc exec vitest bench --run compare` 复现（decimal.js / mathjs 仅为基准对照的 devDependencies，不进运行时）。

## 吞吐基准

下面是 `pnpm --filter @wzo/calc bench`（基于 [Vitest Bench](https://vitest.dev/guide/features.html#benchmarking)）的实测结果，数值因机器而异，仅供量级参考：

| 操作 | 吞吐（ops/s）|
| :--- | ---: |
| `addStr('0.1', '0.2')` | ~8,900,000 |
| `mul('1.1', '2.2')` | ~8,100,000 |
| `chainAdd(10).sub(3).mul(2)()` | ~7,100,000 |
| `add(0.1, 0.2)`（→ number）| ~4,700,000 |
| `fmt(n, { decimals: 2, thousands: true })` | ~2,900,000 |
| `div('1', '3')`（50 位）| ~2,630,000 |
| `calc('1 + 2 * 2')` | ~2,400,000 |
| `calc('9.9 * 3', { _fmt })`（求值 + 格式化）| ~2,080,000 |
| `fmt(n, { compact: 'zh' })` | ~630,000 |
| `calc('clamp(max(3, 5) * 2, 0, 100)')`（数学函数）| ~630,000 |
| —— 对照：原生 `0.1 + 0.2` | ~50,000,000（但 = `0.30000000000000004`）|

> 原生浮点约快 10～15 倍，但结果是错的。`@wzo/calc` 用这点速度换来**严格正确**——单次运算仍在微秒级。

## 正确性对比

这才是 `@wzo/calc` 的价值所在 —— 原生浮点在这些经典场景全错：

| 表达式 | 原生 JS | `@wzo/calc` |
| :--- | :--- | :--- |
| `0.1 + 0.2` | `0.30000000000000004` ❌ | `"0.3"` ✓ |
| `0.3 - 0.1` | `0.19999999999999998` ❌ | `"0.2"` ✓ |
| `0.1 * 0.2` | `0.020000000000000004` ❌ | `"0.02"` ✓ |
| `(1.005).toFixed(2)` | `"1.00"` ❌ | `"1.01"` ✓（`rounding: 'round'`）|
| `9007199254740993 + 0` | `9007199254740992` ❌ | `"9007199254740993"` ✓ |

::callout{type="info"}
基准可复现：在仓库根目录跑 `pnpm --filter @wzo/calc bench`，用例见 `packages/main/tests/perf.bench.ts`。
::

## 几个性能提示

- **精度敏感的值传字符串**：`addStr('0.1', '0.2')` 比 `add(0.1, 0.2)` 略快，且不经过 `Number()`，更安全。
- **格式化字段越少越快**：`compact` / `clamp` 等比纯 `decimals` 略重。
